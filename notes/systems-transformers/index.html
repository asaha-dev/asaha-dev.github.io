If Long Context Isn't Memory, Why Not Just Compress It?

This is a continuation of a series of posts around LLM systems and how the industry is exploring harness engineering around transformers.

In a previous post , I discussed how long context should not be conflated with other system components like memory, retrieval etc. A context window is working memory. Memory is what survives after the context window is gone.

A natural follow-up question is:

If context eventually becomes too large, why don't we just compress it automatically?

At first glance, this seems obvious. Summarize old context, keep the important bits, and continue. In practice, this turns out to be one of the hardest problems in agent architecture.

The transformer architecture was designed for working memory

Part of the answer starts with the transformer itself. Transformers are fundamentally working-memory systems.

The original transformer architecture abandoned recurrent state in favor of attention over an explicitly provided sequence. Instead of carrying hidden state forward indefinitely, the model receives a working set of tokens and computes attention relationships across that working set.

This tradeoff was enormously successful making parallel training possible and scaling improving dramatically leading to large language models emergence. But there was a consequence - the long-term state became somebody else's problem.

The transformer answers:

Given everything currently available, what should I pay attention to?

It does not answer:

What should I remember forever?

The architecture deliberately avoids that question. As context windows grew from thousands of tokens to hundreds of thousands and eventually millions, it became tempting to treat larger context as a substitute for memory. However, a larger working set is still a working set.

Compression sounds easy until you have to throw something away

Suppose an agent has accumulated 100,000 tokens of conversation history.

A simple solution would be:

100,000 tokens
      ↓
summarize
      ↓
2,000 tokens

However, the difficulty is not generating the summary and deciding what to discard. Good compression requires predicting which pieces of information will matter later, often long before the future task is known.

Imagine a conversation containing:

User is allergic to peanuts.

At the moment, this fact may seem irrelevant. Hours later, the user asks for dinner recommendations. A summary that omitted the allergy is now wrong - the problem is that future relevance is unknowable. We humans struggle with this too. We routinely forget details that later become important.

The difference is that people tolerate imperfect memory. Agent systems are often expected to be lossless.

Most context is not one thing

Another challenge is that conversations contain several different categories of information.

Some information is state:

Task 7 completed.
Task 8 in progress.

Some is memory:

User prefers markdown reports.

Some is evidence:

Stack trace
Log output
Research paper

Some is instruction:

Use Python instead of R.

Some is temporary reasoning:

Hypothesis A failed.

These categories should not be compressed in the same way.

A single conversation summary flattens all of them into one representation.

That often destroys exactly the structure the agent needs later.

The challenge is not compression.

The challenge is memory formation.

The system must decide what belongs in state, what belongs in memory, what belongs in storage, and what can safely disappear.

Why /clear often helps

This is why many Claude Code users discover something surprising:

Sometimes the best thing you can do is type:

/clear

At first glance, this seems irrational.

If more context is always better, deleting context should make the system worse.

Yet many practitioners report the opposite.

The reason is that context is not free.

As conversations grow, the working set accumulates stale plans, abandoned hypotheses, obsolete instructions, dead ends, temporary reasoning, and partially completed ideas.

Even if the model can technically access all of this information, it still competes for attention.

The problem is not that the model forgot something.

The problem is that it is trying to consider too many things simultaneously.

/clear works because it discards working memory.

The next session starts with a smaller and cleaner working set.

This is not evidence that memory is unnecessary.

It is evidence that working memory and memory serve different purposes.

Good memory systems make forgetting safe.

The hidden role of the harness

This observation leads directly to one of the most important shifts happening in agent design.

Increasingly, the intelligence is not living entirely inside the model.

It is living in the harness around the model.

Anthropic's work on long-running agents is a good example.

Their research repeatedly emphasizes artifacts such as:

  • progress files
  • task lists
  • handoff documents
  • checkpoints
  • git history
  • structured state

These artifacts allow work to continue across sessions even when the context window is reset.

The model does not remember what happened.

The harness does.

Viewed this way, a progress file is a form of episodic memory.

A task tracker is a state-management system.

A code repository is an external memory store.

The harness is increasingly responsible for deciding what information survives and what gets reconstructed later.

Why harness engineering is becoming a field

This is one reason harness engineering is emerging as its own discipline.

Early LLM applications focused almost entirely on prompts and models.

Modern agent systems increasingly focus on everything around the model:

  • retrieval
  • memory formation
  • state management
  • checkpointing
  • tool discovery
  • task decomposition
  • execution monitoring

These systems exist because larger context windows do not solve all continuity problems.

In many cases they simply postpone them.

The practical challenge is no longer:

How do I fit more information into context?

The practical challenge is:

What information deserves to survive after context is gone?

That question sits at the intersection of memory, retrieval, state management, and learning.

It is increasingly where agent architecture is happening.

The emerging architecture

One way to view the history of language models is that transformers solved working memory extraordinarily well.

Everything else is now being rebuilt around them.

Context windows provide active working memory.

Retrieval systems provide access.

Memory systems decide what persists.

State-management systems track ongoing work.

Harnesses coordinate the entire process.

The result is starting to look less like a single model with a giant context window and more like a complete cognitive architecture assembled from multiple interacting components.

The irony is that as context windows get larger, the importance of these external systems appears to be increasing rather than decreasing.

The problem was never storing more tokens.

The problem was deciding which tokens matter.
