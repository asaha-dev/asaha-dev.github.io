The latest buzzword has been around harness engineering. Context windows keep getting larger, but the investment in scaffolding around the model keeps growing alongside them. Here's my readout to explain why.

Part of a series on LLM systems: building on context and memory, this post looks at why larger context windows don't make harness engineering obsolete — compression, state formation, and external scaffolding are where continuity actually gets solved. Claude Code deep dive.

As context windows expand, more effort is being invested in memory formation, retrieval, state management, checkpointing, task orchestration, and harness engineering. Claude Code's /clear and /compact commands are a small but concrete example of this trend. They exist because managing information turns out to be a harder problem than simply storing it. Understanding why requires looking at how transformers handle context, why compression is fundamentally difficult, and why increasingly sophisticated systems are being built around the model itself.

In a previous post, I discussed how context, retrieval, memory, and learning are distinct systems that are frequently conflated. A context window is working memory. Memory is what survives after the context window is gone.

A natural follow-up question is:

If context eventually becomes too large, why don't we just compress it automatically?

At first glance, this seems obvious. Summarize old context, keep the important bits, and continue. In practice, this turns out to be one of the hardest problems in agent architecture.

The transformer architecture was designed for working memory

Part of the answer starts with the transformer itself. Transformers are fundamentally working-memory systems.

The original transformer architecture abandoned recurrent state in favor of attention over an explicitly provided sequence. Instead of carrying hidden state forward indefinitely, the model receives a working set of tokens and computes attention relationships across that working set.

This tradeoff was enormously successful — parallel training became possible, scaling improved dramatically, and large language models emerged. But there was a consequence: long-term state became somebody else's problem.

The transformer answers:

Given everything currently available, what should I pay attention to?

It does not answer:

What should I remember forever?

The architecture deliberately avoids that question. As context windows grew from thousands of tokens to hundreds of thousands and eventually millions, it became tempting to treat larger context as a substitute for memory. However, a larger working set is still a working set.

Compression sounds easy until you have to throw something away

Suppose an agent has accumulated 100,000 tokens of conversation history.

A simple solution would be:

100,000 tokens
      ↓
summarize
      ↓
2,000 tokens

However, the difficulty lies not in generating the summary but in deciding what to discard. Good compression requires predicting which pieces of information will matter later, often long before the future task is known.

Imagine a conversation containing:

User is allergic to peanuts.

At the moment, this fact may seem irrelevant. Hours later, the user asks for dinner recommendations. A summary that omitted the allergy is now wrong - the problem is that future relevance is unknowable. We humans struggle with this too. We routinely forget details that later become important. The difference is that people tolerate imperfect memory. Agent systems are often expected to be lossless.

Most context is not one thing

Another challenge is that conversations contain several different categories of information. Some information is state:

Task 7 completed.
Task 8 in progress.

Some is memory:

User prefers markdown reports.

Some is evidence:

Stack trace
Log output
Research paper

Some is instruction:

Use Python instead of R.

Some is temporary reasoning:

Hypothesis A failed.

These categories should not be compressed in the same way. A single conversation summary flattens all of them into one representation, which often destroys exactly the structure the agent needs later. The challenge, then, is not compression but memory formation. The system must decide what belongs in state, what belongs in memory, what belongs in storage, and what can safely disappear.

Why /clear and /compact exist

Claude Code ships with two commands for managing context. /clear discards the working memory entirely and starts fresh. /compact summarizes the conversation and continues with a compressed version.

Both exist for the same underlying reason: context accumulates noise faster than signal. As conversations grow, the working set fills with stale plans, abandoned hypotheses, obsolete instructions, dead ends, temporary reasoning, and partially completed ideas. Even when the model can technically access all of it, the information competes for attention. The problem is not that the model forgot something. The problem is that it is trying to consider too many things simultaneously.

Claude Code does also run automatic compaction. An auto-compact stage fires when context pressure reaches a threshold — the pipeline summarises without waiting for user input, because the alternative is the model running out of context mid-task. This is the technical floor: when the window would otherwise overflow, the system handles it.

The user commands exist for a different purpose. /compact lets you trigger compaction deliberately, before the forced threshold, and verify that the summary captured what matters. /clear discards everything when accumulated context is more noise than signal and a clean start is worth the loss of history. The choice between them comes back to the compression problem: the system cannot reliably predict which parts of the conversation the user still needs. Auto-compact makes a reasonable guess when forced to; the user commands give agency over the decision when there is time to make it intentionally.

Anthropic's guidance roughly maps to: use /compact when you want to continue the current task but the conversation has grown unwieldy; use /clear when you want a fresh start. The distinction — automatic when the window forces it, user-controlled when chosen — reflects that compaction is sometimes a technical necessity and sometimes a judgment call. Good memory systems make forgetting safe — but knowing what to forget still requires judgment that the user, for now, is better placed to exercise.

The hidden role of the harness

This observation leads directly to one of the most important shifts happening in agent design. Increasingly, the intelligence is not living entirely inside the model. It is living in the harness around the model. Anthropic's work on long-running agents is a good example. Their research repeatedly emphasizes artifacts such as:

  • progress files
  • task lists
  • handoff documents
  • checkpoints
  • git history
  • structured state

These artifacts allow work to continue across sessions even when the context window is reset. The model does not remember what happened - the harness does. Viewed this way:

  • A progress file is a form of episodic memory.
  • A task tracker is a state-management system.
  • A code repository is an external memory store.

The harness is increasingly responsible for deciding what information survives and what gets reconstructed later.

Why harness engineering is becoming a field

This is one reason harness engineering is emerging as its own discipline. Early LLM applications focused almost entirely on prompts and models. Modern agent systems increasingly focus on everything around the model:

  • retrieval
  • memory formation
  • state management
  • checkpointing
  • tool discovery
  • task decomposition
  • execution monitoring

These systems exist because larger context windows do not solve all continuity problems. In many cases they simply postpone them.

The practical challenge is no longer:

How do I fit more information into context?

The practical challenge is:

What information deserves to survive after context is gone?

That question sits at the intersection of memory, retrieval, state management, and learning. It is increasingly where agent architecture is happening.

The emerging architecture

One way to view the history of language models is that transformers solved working memory extraordinarily well. Everything else is now being rebuilt around them. Context windows provide active working memory; retrieval systems provide access; memory systems decide what persists; state-management systems track ongoing work; harnesses coordinate the entire process.

The result is starting to look less like a single model with a giant context window and more like a complete cognitive architecture assembled from multiple interacting components.

As context windows get larger, the importance of these external systems appears to be increasing rather than decreasing. This is why harness engineering is gaining popularity — it is where the continuity problem is actually being solved.
