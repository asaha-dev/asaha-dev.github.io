I've been thinking through the shift from ReAct loops to engineered agent harnesses. ReAct is still a useful mental model, but the loop by itself no longer explains where production agent systems succeed or fail.

This is the first post in a series on LLM systems: why the industry is moving away from "just give the model more context" and toward explicit memory formation, orchestration systems and harness engineering. The series goes deeper in context and memory, building systems around transformers, and Claude Code: a systems architecture deep dive.

The core architectural change is that the reasoning loop is no longer the whole system. It is one component inside a harness: a deterministic orchestration layer around a non-deterministic model that explicitly owns context, control flow, state, tools, and verification.

ReAct was widely popular in 2022-2023 for building AI applications on large language models. The core idea was neat: let the model interleave reasoning traces with tool calls in one flat loop. Instead of forcing the LLM to either think internally and answer (User → LLM → Answer) or call tools immediately (User → LLM → Tool → LLM → Answer), you alternate between thought, action, and observation, possibly repeating, before arriving at an answer.

For example:

Question:
Who is the current CEO of Nvidia?

Thought:
I need to find who the CEO is.

Action:
Search("CEO Nvidia")

Observation:
Jensen Huang is the founder and CEO of Nvidia.

Thought:
I found the answer.

Answer:
Jensen Huang

This became the blueprint for most modern agents. ReAct became the dominant pattern for early LangChain agents and tool-calling implementations. LangGraph later emerged as a more structured alternative, adding explicit state and control flow in response to the flat loop's limitations at scale.

An agent harness is the systems idea that wraps it. The harness is what you build around the model for a specific task. Frameworks like LangChain, LangGraph, CrewAI, or other SDKs are tooling that helps you build a harness, or provides one off the shelf. The distinction matters because the harness is where context, state, verification, and recovery decisions live.

A useful canonical harness loop is gather context → take action → verify work → repeat. The verify stage is a meaningful addition that classic ReAct does not have: it externalises judgment rather than asking the model to self-critique.

I find the transition easiest to reason about through three lenses:

lens 1: architecture and design: where the flat loop breaks

A pure ReAct loop dumps every thought, action, and observation into one growing context window. That works surprisingly well for short tasks. It starts to fail when the loop becomes the only place where state lives.

The failure mode is not simply that the model forgets. The context window fills with stale plans, abandoned hypotheses, intermediate outputs, tool traces, and partial decisions. Even when the model can technically see the tokens, the relevant state has to compete with everything else.

The architecture that addresses this usually separates three things:

session: an append-only event log that survives failures and restarts.
harness: the orchestration layer that decides what context to load, what action to take, and how to verify it.
sandbox: the execution boundary for tools, shell commands, browser checks, code execution, or remote environments.

This split matters because the flat loop has no durable place to put truth. Imagine an agent refactoring a codebase. It investigates three approaches, abandons one after tests fail, edits several files, and then gets compacted. If all of that lives only in the conversation, the next model call has to infer current state from a compressed narrative. The summary may preserve the final decision but lose why an earlier approach failed. It may preserve the todo list but lose which tests already ran.

In a harnessed system, those categories separate. The repository records the actual state of the work; a progress file or task list records what remains; and the session log records what happened. The sandbox provides the execution boundary. The model can still reason over all of this, but it is no longer responsible for being the only state store.

Q1. What design choices are critical?

a. State. Compaction alone is not enough because state lives in artifacts, not in the context window. Progress files, task lists, session logs, repositories, and test results give the system durable places to put truth.

b. Evaluation. Separate the doer from the judge. A standalone evaluator is more tunable than asking the generator to validate its own work. Rules-based verification, including linting, tests, and visual checks, still matters.

lens 2: build and framework choices: what to own directly

The question of which framework to use is real, but it is downstream of a more important one: how much of the harness do you want to own directly? A framework gives you pre-built abstractions, such as graph nodes, agent roles, tool connectors, and memory stores, in exchange for structural decisions made on your behalf. When those decisions match your task, frameworks accelerate. When they don't, they add layers to debug between you and the loop.

The build-your-own-first philosophy resonates with me here. Start close to the API when the patterns are simple, and use a framework when you understand enough of the loop to know which complexity it is taking off your plate.

Q1. How do I choose an appropriate framework?

Use a raw loop and direct API when you want to understand the harness and the patterns are simple. Use a framework or SDK when you want harness concerns solved for you: automatic compaction, sub-agent context isolation, permissions, tool execution, or first-class verification. Use LangGraph when you need explicit graph or state-machine control over deterministic orchestration.

The risk shows up when the framework owns the part of the system where the product differentiates: what context gets loaded, what state survives, how work is verified, how failures recover, and how evals map to user trust. Those are not incidental implementation details. They are often the product behavior.

Q2. What should I avoid outsourcing?

The parts of the loop where the product differentiates: how context gets curated, what state survives, how work is verified, how failures recover, and how evals map to user trust. The framework is not the moat; the harness, context curation, and evals are.

lens 3: strategy and organization: how to make decisions

Some of the questions that come up in practice:

Q1. Where does durable advantage live?

The model is often rented. What compounds inside the org is the session log, the harness, and the eval infrastructure. The session log preserves what happened. The harness encodes how the system works. The evals tell you when a model upgrade helps or hurts your task.

Q2. Build vs buy

If starting today, I would build a thin direct-API wrapper first to learn where the tasks actually break. Then I would adopt a framework only once I understand the loop well enough to know what it's hiding from me. The org's capability to build the harness is the asset. Outsourcing the harness outsources the most valuable part.

Q3. How to frame the transition to stakeholders

Most people who are not deep in this space perceive an LLM as one unit: you ask a question, you get an answer. The reframe that tends to work is: "we're moving from asking the model to do everything, to engineering the conditions under which it succeeds." That shifts the conversation from AI capability, which many stakeholders cannot evaluate, to system design, which they can.

Q4. Which pattern for which job?

The field debate between single and multi-agent could not be louder. However, my read is that the strongest arguments on both sides are not actually contradicting each other. They are describing different tasks. The productive question is not which camp is right but which pattern fits the work.

Use a single-agent pattern for dependency-heavy, coherent-output work: code, documents, anything where later steps depend on earlier ones. Use multi-agent patterns for parallelisable, breadth-first work: research, competitive analysis, broad search, synthesis. Design the interfaces before the agents so the pattern choice remains reversible.

Does the model eat the harness?

The strongest version of the critique is that tool-calling systems, orchestration layers, and other scaffolding have a short shelf life before capabilities migrate into the model itself. Some of that is clearly true. Workarounds written for weaker models can become dead weight as the model improves.

That does not make the harness irrelevant. It changes what kind of harness work is worth doing.

Model-specific scaffolding decays quickly but interfaces decay more slowly. A workaround for a weak model may become obsolete with the next release, but a clean separation between session state, sandbox execution, verification, and context loading still gives the system room to evolve.

That is my direction: build the harness so the unstable parts can be replaced. Keep the session and sandbox interfaces stable. Keep verification explicit. Keep evals close to the decisions they support. Let the orchestration logic change as model capability changes.
