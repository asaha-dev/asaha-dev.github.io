What I was reading

This is the first post in a series on LLM systems: why the industry is moving away from "just give the model more context" and toward explicit memory formation and orchestration systems. The series goes deeper in context and memory, building systems around transformers, and Claude Code — a systems architecture deep dive.

ReAct (Yao et al., 2022) was widely popular in 2022–2023 for building AI applications on large language models. The core idea was neat: let the model interleave reasoning traces with tool calls in one flat loop. Instead of forcing the LLM to either think internally and answer (User → LLM → Answer) or call tools immediately (User → LLM → Tool → LLM → Answer), you alternate between thought, action, and observation — possibly repeating — before arriving at an answer. For example:

Question:
Who is the current CEO of Nvidia?

Thought:
I need to find who the CEO is.

Action:
Search("CEO Nvidia")

Observation:
Jensen Huang is the founder and CEO of Nvidia.

Thought:
I found the answer.

Answer:
Jensen Huang

This became the blueprint for most modern agents. ReAct became the dominant pattern for early LangChain agents and tool-calling implementations. LangGraph later emerged as a more structured alternative — adding explicit state and control flow in response to the flat loop's limitations at scale.

An agent harness is the systems idea that wraps it: a deterministic orchestration layer around a non-deterministic model that explicitly owns context, control flow, and verification. We are not replacing the reasoning loop — we are demoting it from the whole system to one component inside a scaffold that we engineer. The harness is what you build, specific to your task. A framework — LangChain, LangGraph, CrewAI — is tooling that helps you build a harness, or provides one off the shelf. The two are often conflated, and the difference matters more than it might seem at first.

Anthropic's Claude Agent SDK frames the canonical harness loop as gather context → take action → verify work → repeat. The verify stage is a meaningful addition that classic ReAct does not have — it externalises judgment rather than asking the model to self-critique.

I explored this transition through three lenses:

Lens 1 — Architecture / design

Why the single loop breaks at scale

A pure ReAct loop dumps every thought, action, and observation into one growing context window. This triggers what Anthropic's context engineering research calls "context rot" — recall accuracy degrades as token count climbs, because the transformer's attention budget is finite. A few reference points on the scale of this problem:

  • Laban et al.'s "LLMs Get Lost In Multi-Turn Conversation" (2025) measured an average 39% performance drop across six generation tasks in multi-turn versus single-turn settings, across 200,000+ simulated conversations. The mechanism is specific: models make assumptions in early turns, prematurely commit to a direction, and then fail to recover when that direction proves wrong — a flat loop has no structure to interrupt this.
  • Stanford's "lost in the middle" research (Liu et al., 2023) documents a related failure mode: performance degrades significantly when relevant information appears in the middle of long contexts, even in models that technically process all tokens.
  • Anthropic documented the concrete failure mode in practice: the model runs out of context mid-task and leaves work half-implemented, or a later instance prematurely declares the job done.

The harness patterns that address this are fundamentally context-management moves:

  • Context engineering — runtime curation of the optimal token set during inference; a discipline distinct from prompt engineering
  • Compaction — summarises a near-full window, reinitialises fresh
  • Structured note-taking — persists state outside the window: progress files, JSON feature lists, git log
  • Sub-agent isolation — sub-agents explore in clean windows, return only a ~1–2K token distilled summary
  • Verification/critic loop — a separate evaluator agent judges the work
  • Workflow decomposition — predefined code paths instead of one autonomous loop

The concrete architecture that makes these patterns work is a three-component split: session (an append-only event log outside the context window — the thing that survives failures and restarts), harness (a stateless orchestrator that calls the model and encodes no model-specific assumptions), and sandbox (ephemeral execution environments provisioned as tools, called via a simple execute(name, input) → string interface). The harness calls the sandbox; the sandbox returns a string. Keeping the harness stateless and the interfaces stable is what allows the orchestration logic to evolve as model capabilities do without rewriting the whole system.

Two findings worth noting for anyone in a decision-making seat:

  1. Compaction alone is not sufficient — the context and memory and building systems around transformers posts cover this in detail. The short version: state lives in artifacts, not in the context window.
  2. Separate the doer from the judge — there is a well-documented self-preference bias (arXiv 2404.13076, NeurIPS 2024). A standalone evaluator is more tunable than a self-critical generator. Anthropic's long-running harness uses a planner + generator + evaluator pattern running multi-hour sessions, with the evaluator driving a browser to test like a user. Also worth noting: rules-based verification (linting, tests, visual checks) should not be discarded in favour of LLM-as-judge. The latter enables novel verification pathways but is less robust and higher latency.

Lens 2 — Build / framework choices

The question of which framework to use is real, but it is downstream of a more important one: how much of the harness do you want to own directly? A framework gives you pre-built abstractions — graph nodes, agent roles, tool connectors, memory stores — in exchange for structural decisions made on your behalf. When those decisions match your task, frameworks accelerate. When they don't, they add layers to debug between you and the loop.

I did not find comparative claims against LangGraph, OpenAI Agents SDK, CrewAI, Google ADK, or AutoGen that survived adversarial verification — most benchmarks are vendor-produced or task-specific.

Anthropic's building effective agents post articulates a build-your-own-first philosophy worth quoting directly:

We suggest that developers start by using LLM APIs directly: many patterns can be implemented in a few lines of code. If you do use a framework, ensure you understand the underlying code. Incorrect assumptions about what's under the hood are a common source of customer error.

They name five composable workflow patterns as structured alternatives to a monolithic loop: prompt chaining, routing, parallelisation, orchestrator-workers, and evaluator-optimizer. Most teams need these, not a full autonomous agent.

A rough decision framework I use:

  • Raw loop + direct API — when I want to understand the harness and the patterns are simple
  • Framework like Claude Agent SDK or others — when I want harness concerns solved for me: automatic compaction, sub-agent context isolation, first-class verification. Use Claude Agent SDK if your org is already Claude-centric; otherwise consider OpenAI Agents SDK, CrewAI, AutoGen, or Google ADK depending on whether role/crew abstractions or cloud-native integration matter more.
  • LangGraph — when I need explicit graph/state-machine control over deterministic orchestration and the most control over execution paths

The lesson here is that the framework is not the moat — it is one way to build toward one. The harness, context curation, and evals are. A framework that hides the loop from you is a liability exactly because the loop is where the differentiation lives: how you manage context, what you verify, how you recover from failure. You can build that on top of LangGraph or on top of raw API calls. What you cannot do is outsource it.

Lens 3 — Strategic / org

Some of the questions that come up in practice:

Where does durable advantage live? The model is often rented, a rapidly commoditising input — everyone gets the next frontier model if they're willing to pay for it. What compounds inside the org is more concrete than it sounds: the session log (durable, append-only, the thing that survives model failures and harness restarts), the stateless harness (orchestration logic that can evolve without being coupled to a specific model's behaviour), and the eval infrastructure (the thing that tells you when a model upgrade helps or hurts your task). Anthropic's own managed-agents work found that decoupling these three components dropped time-to-first-token roughly 60% at the median and over 90% at the 95th percentile — not because the model got faster, but because the architecture stopped paying costs it didn't need to. Invest the team's engineering hours in these three things, not in chasing model-level optimisations that the next frontier release will absorb anyway.

Build vs buy If starting today, I would build a thin direct-API wrapper first to learn where the tasks actually break — then adopt a framework only once I understand the loop well enough to know what it's hiding from me. The org's capability to build the harness is the asset. Outsourcing the harness outsources the most valuable part.

How to frame the transition to stakeholders Most people who are not deep in this space perceive an LLM as one unit — you ask a question, you get an answer. Proposing a harness system can come across as overkill to anyone not watching what's happening beneath the ChatGPT or Claude interface. The narrative "we're moving from clever prompts to engineered systems" is hard to land cold.

The reframe that tends to work: "we're moving from asking the model to do everything, to engineering the conditions under which it succeeds." This shifts the conversation from AI capability — which most stakeholders can't evaluate — to system design, which they can. A useful analogy to use: the model is the engine. We're building the car. A powerful engine without a drivetrain, steering, and brakes isn't a vehicle you can use in production.

Which pattern for which job?

The field debate between single and multi-agent could not be louder. However, my read is that Cognition's "Don't build multi-agents" and Anthropic's life sciences system are not contradicting each other — they are describing different tasks. The productive question is not which camp is right but which pattern fits the work.

Use single-agent for dependency-heavy, coherent-output work. Code, documents, anything where later steps depend on earlier ones. Cognition's finding was precise: the failure mode is not multi-agent in general, it is parallel agents making conflicting implicit assumptions about shared output. A reviewer agent with fresh context is fine and often valuable — parallel writers on the same codebase are not.

Use multi-agent for parallelisable, breadth-first work that exceeds one context window. Research, competitive analysis, broad search, synthesis — tasks where outputs are independent and combined at the end. Anthropic's life sciences system (outperforming human baseline on Protocol QA with connectors to Benchling, PubMed, BioRender, and 10x Genomics) and Google's AI co-scientist ("generate, debate, evolve" across biomedical hypothesis generation, independently discovering a novel gene transfer mechanism that matched unpublished experimental results) are both in this category. The task structure justifies the coordination overhead.

Design the interfaces before the agents. This is the recommendation that makes the pattern choice reversible. With stable session / harness / sandbox interfaces — the harness calls the sandbox via a defined contract, the session log holds state outside the context window — you can add a reviewer agent, switch from single to multi, or change the execution environment without rewriting the system. Without stable interfaces, each architectural change becomes a rewrite.

Consider the cost. Anthropic's multi-agent research system is direct about the spend:

In practice, these architectures burn through tokens fast. In our data, agents typically use about 4× more tokens than chat interactions, and multi-agent systems use about 15× more tokens than chats.

Run the expensive pattern only where the task genuinely justifies it.

Some additional insights from the industry.

A more fundamental challenge to the harness thesis comes from a different direction than the single-vs-multi debate.

Logan Kilpatrick, in a Sequoia Capital podcast (June 2025), argued that the scaffolding being built around models — tool-calling systems, agent harnesses, orchestration layers — has roughly a 12-month shelf life before capabilities migrate natively into model weights. The model eats the harness. What gets built today becomes commoditised into the next frontier release. The definition of "the model" has already expanded to encompass tool calling, containers, and agent harness functions that were external only a couple of years ago; Kilpatrick's argument is that this expansion continues.

Kangwook Lee makes a more structural version of the same point. He proposes three zones of model capability: below a threshold, models cannot do the task regardless of harness quality; in the middle zone, the harness is the deciding variable; above the threshold, models are capable enough that human-designed harnesses become the bottleneck rather than the enabler. His group showed that for mid-sized models, AI-designed harnesses already outperform human-designed ones through iterative engineering. If that generalises upward, human harness engineering becomes a constraint at exactly the moment when model capability makes the constraint most visible.

My thinking is that that both arguments are right about the direction; the timing to be seen. Anthropic's engineering confirms part of the premise; their post on managed agents work describes finding that harness workarounds written for earlier Claude models became dead weight as the model improved; scaffolding built around "context anxiety" in older versions was eliminated by a single model upgrade. Their design response was not to stop building harnesses, but to make the harness encode no model-specific assumptions — opinionated about the interfaces (session manipulation, sandbox execution) but agnostic about implementation. The session and sandbox interfaces stay stable; the harness logic is free to change. That is the more actionable response to the critics: build for interface stability, not implementation permanence.

The architectural choices in Lens 1 and 2 — which context management patterns to adopt, how to structure verification, which framework to use — are not just engineering decisions. They determine what your org can build, how fast you can iterate on evals, and whether you own the most valuable layer of the stack or rent it.
