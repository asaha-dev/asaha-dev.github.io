Claude Code is one of the most interesting production agent systems to study right now. I went through the VILA Lab reverse-engineering analysis to understand how it's actually built.

Part of a series on LLM systems: a concrete look at how one production agent system implements the architectural principles described in context and memory and building systems around transformers.

Claude Code is built around a deceptively simple core: an async generator calls the model, inspects the response for tool calls, executes them, feeds results back, and repeats. When the model returns plain text without a tool invocation, the loop terminates. The surface area of this core loop is small. Most of the engineering lives in the five layers of infrastructure built around it.

the five-layer stack

The architecture organizes into five distinct layers, each with a defined responsibility:

┌────────────────────────────────────────────────────────┐
│                    SURFACE LAYER                        │
│       CLI  ·  Agent SDK  ·  IDE  ·  Browser            │
└─────────────────────────┬──────────────────────────────┘
                          │  all surfaces converge on one loop
┌─────────────────────────▼──────────────────────────────┐
│                     CORE LAYER                          │
│       queryLoop()  ·  five-stage compaction pipeline    │
└─────────────────────────┬──────────────────────────────┘
                          │  model reasons · harness enforces
┌─────────────────────────▼──────────────────────────────┐
│              SAFETY / ACTION LAYER                      │
│  permissions  ·  ML classifier  ·  27-event hooks       │
│  tool pool assembly  ·  subagent spawning  ·  sandbox   │
└─────────────────────────┬──────────────────────────────┘
                          │  durable, auditable state
┌─────────────────────────▼──────────────────────────────┐
│                    STATE LAYER                          │
│    append-only JSONL  ·  CLAUDE.md hierarchy            │
│    auto-memory  ·  sidechain subagent transcripts       │
└─────────────────────────┬──────────────────────────────┘
                          │  execution surface
┌─────────────────────────▼──────────────────────────────┐
│                   BACKEND LAYER                         │
│      tool implementations  ·  MCP servers               │
│      filesystem  ·  shell  ·  remote execution          │
└────────────────────────────────────────────────────────┘

Every entry point — CLI, SDK, IDE extension, browser — routes through the same queryLoop() function. There is no separate mode-specific engine per surface. The uniform path ensures that behavior is consistent regardless of how the agent is invoked.

core layer: the loop and context management

The queryLoop() async generator is the central control structure. It is single-threaded, with one flat message history. This design explicitly avoids threaded conversations or multiple competing agent personas, which introduce unpredictable behavior at scale.

The other critical component of the core layer is the five-stage compaction pipeline, which runs before every model call. rather than official Anthropic documentation, the stages — budget reduction, snipping, micro-compaction, context collapse, and auto-compact — apply in sequence as context pressure increases. The context window is the binding constraint in transformer-based agentic systems: the attention mechanism scales quadratically with sequence length, and model performance degrades measurably as the window fills, a phenomenon Anthropic refers to as "context rot." The compaction pipeline is the architectural response to this constraint — the equivalent of a cache eviction policy applied to token sequences.

The "just-in-time" context loading strategy complements compaction. Rather than pre-loading all relevant information upfront, the agent maintains lightweight identifiers — file paths, query handles, URLs — and loads content on demand. This avoids paying the context cost of material that the task may never require.

safety / action layer: separation of reasoning and enforcement

The safety layer contains seven independent mechanisms: pre-filtering of denied tools before model invocation, deny-first rules that override allow rules, permission modes that define baseline behavior, an ML classifier for automated evaluation, shell sandboxing for execution restriction, non-restoration of permissions on session resume, and hooks that intercept throughout. The layering is deliberate — each mechanism assumes the prior one can fail.

The placement of the ML classifier is architecturally significant. It operates at the output layer: after the model produces an action and before that action executes. The model reasons without input filtering; the harness constrains what actions reach execution. This separation ensures that the model sees the full distribution of tasks, while enforcement remains outside the model's control.

The hook system provides 27 interception points across the tool execution lifecycle (again, per VILA Lab's analysis). This is the sidecar pattern applied to agent actions — external logic can observe and modify behavior without being embedded in the core loop.

state layer: append-only durability

Session transcripts are stored as append-only JSONL files. Nothing is mutated in place. This produces the same properties as write-ahead logging in databases: auditability, crash recovery, and the ability to reconstruct state from the log. Checkpoint and rollback are straightforward to implement as pointer operations on the log rather than snapshot operations.

Subagent transcripts are stored in separate sidechains, isolating their context from the parent. This is the same principle as microservice state isolation: parent-child coupling via shared mutable state creates unpredictable failure modes.

The CLAUDE.md hierarchy — org-level, user-level, project-level, local — provides version-controllable, file-visible configuration rather than opaque database storage. This is consistent with the broader design principle of preferring externalized, inspectable policy over internal state.

extensibility: four mechanisms at different context costs

Four extension mechanisms operate at different layers: MCP servers for external integrations, plugins for distributable component bundles, skills for injecting domain-specific instructions, and hooks for lifecycle interception. The choice of four mechanisms rather than a unified API reflects a deliberate trade — each mechanism has a different context cost and a different deployment model, and collapsing them would either over-constrain low-cost extensions or under-constrain high-cost ones.

MCP is the closest analogue to a service mesh. It provides a standardized protocol with typed interfaces, multiple transport variants (stdio, HTTP, SSE), and schema-defined capabilities. External tools become first-class citizens in the tool pool, routed through a single protocol rather than ad-hoc integrations. The 2024 MCP specification did for agent extensibility what Protocol Buffers did for distributed service interfaces — established a common schema language for capability description.

what the architecture deliberately omits

What I find most revealing about this architecture is what it leaves out. There are no explicit planning graphs, no developer-defined state machines, no scaffolding frameworks that constrain the model's decision sequence. The investment is in the operational harness — context management, tool routing, safety enforcement, state durability — rather than in decision scaffolding. The premise is that capable models benefit more from a rich operational environment than from frameworks that constrain their choices.

Explicit scaffolding limits the model's ability to adapt its approach to novel situations. Choosing strong invariants at the boundaries — safety, permissions, state durability — with freedom inside them is a different approach to reliability than constraining the decision space.

The context window as a first-class resource, the append-only state model, the layered safety architecture, the separation of model reasoning from harness enforcement — these choices produce a system whose failure modes are more predictable and whose operational properties are more inspectable than alternatives that embed policy inside the model or collapse the extension layers into one.
