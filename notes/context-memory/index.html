Recently I have been digging deeper into context engineering and memory systems for agents. Here are some of my notes.

Let's start with a mental model of the different terminologies:

Context answers:
"What is available right now?"

Retrieval answers:
"What can I find?"

Memory answers:
"What should persist?"

Learning answers:
"What changed the system itself?"

Architecturally these are different systems solving different problems. In practice they are frequently conflated, and that conflation tends to produce architectural mistakes that are hard to diagnose.

When a model references something mentioned 50,000 tokens earlier, it has not remembered it in the way we typically think about memory. The information never left the active working set. The model can still attend to those tokens because they remain inside the context window. Nothing was persisted or selectively retained by the system and nothing was recalled from storage. The information was simply never removed.

         CONTEXT WINDOW                   MEMORY SYSTEM
         ──────────────                   ─────────────
         Active working set               Persists beyond a session
         Verbatim retention               Selective retention
         Session-scoped                   Cross-session continuity
         Finite capacity                  Explicit storage layer
         Recomputed access                Retrieved on demand
         Temporary conditioning           Durable state

what context actually does

The transformer architecture provides a form of working memory. Information remains available because it stays inside the active context window. This is extremely useful, but it has limits. I will dig deeper into this aspect in another note.

The scaling problem becomes concrete quickly. Anthropic's context engineering research documents what they call "context rot": model performance degrades measurably as the context window fills, even when the model can technically access all prior tokens. Attention dilutes across longer sequences. The "lost in the middle" problem remains observable even in models with very large context windows, although newer architectures mitigate it to varying degrees. Recency bias remains strong. A 1M token window increases the working set; it does not resolve the retrieval problem within that window.

The cost dimension compounds this. Standard dense transformer attention scales quadratically with sequence length, although modern architectures increasingly employ sparse, local, and approximate attention mechanisms to reduce that cost.

A larger context window increases how much information can be kept active. It does not automatically solve retrieval, prioritization, or memory formation. This distinction becomes visible in production systems.

the MCP lesson

A concrete example comes from MCP tool schemas.

When an agent connects to MCP servers, it often receives tool names, descriptions, parameters, JSON schemas, and usage instructions up front. Large tool collections can consume tens or even hundreds of thousands of tokens before a user asks the first question.

Cost is only part of the problem. Context is also a shared resource. Tool schemas compete with retrieved documents, conversation history, instructions, intermediate reasoning, and task-specific information. Anthropic's engineering work on code execution with MCP documents the scale of this: Jira's MCP schema alone accounts for around 17,000 tokens. Anthropic's internal tooling hit 134,000 tokens of tool definitions before optimization work began. A single workflow connecting Google Drive to Salesforce via MCP consumed 150,000 tokens before code-execution-based deferred loading reduced that to 2,000 — a 98.7% reduction on that task.

Anthropic's tool-search work demonstrates a simple but important idea: load information only when it is needed.

The lesson is broader than tools. Systems frequently improve when irrelevant context is removed rather than when additional context is added. The improvement comes from better retrieval and selective loading, not from expanding the context window.

retrieval is not memory either

It is tempting to replace one misconception with another. If long context is not memory, retrieval is not memory either. A vector database, search index, or retrieval system primarily answers:

What information can I find?

Memory answers a different question:

What information deserves to persist?

Storing every interaction in a vector database creates a searchable archive, but not necessarily a memory system. Memory requires selection, abstraction, retention policies, and retrieval.

A useful shorthand is:

Context   = active working set
Retrieval = access mechanism
Memory    = persistence + abstraction + retrieval

memory is selective retention

Human memory is not a verbatim transcript of experience. What survives is usually an abstraction: a fact, pattern, relationship, lesson, preference, or skill.

The same principle applies to AI systems. A system that stores everything equally is closer to a logging system than a memory system. The hard problem is deciding what deserves to persist, what can be discarded, how information should be represented, and how it can later be retrieved.

Viewed through this lens, most memory architectures are forms of selective representation.

Conversation summaries compress interaction history into reusable state. Episodic memories compress events into records. Knowledge graphs compress information into entities and relationships. Retrieval systems compress large corpora into searchable representations.

The challenge is not storage. Storage is cheap. The challenge is deciding what should survive after the context window is gone.

state is not memory

Another distinction that frequently gets blurred is the difference between memory and state. Memory answers questions about the past — what happened, what was learned, what should be retained. State answers questions about the present — what task is underway, what step completed, what happens next.

Checklists, task graphs, scratchpads, execution plans, and workflow trackers are primarily state-management systems.

Many successful agents combine both. Memory preserves information across interactions. State preserves continuity within an interaction.

remembering is not learning

Long-term memory and learning are often treated as the same phenomenon. They are not. Memory changes what information is available to the system. Learning changes how the system behaves.

A stored user preference affects future decisions because the information can be recalled. A fine-tuned model behaves differently because its parameters changed. The former is persistence. The latter is adaptation.

This distinction is particularly important when discussing procedural memory. Current AI systems primarily acquire durable behavioral changes through training, fine-tuning, reinforcement learning, or other parameter updates.

Context can temporarily simulate skill through instructions, examples, and demonstrations. Once the context disappears, so does the behavior. That is temporary conditioning, not learning.

what the industry is rebuilding

One consequence of the transformer architecture is that long-term state increasingly lives outside the model. Viewed through that lens, much of modern agent architecture can be interpreted as rebuilding capabilities that context windows alone do not provide — retrieval systems for recall, episodic stores to preserve history, memory formation pipelines to decide what should persist, knowledge graphs for relational structure, and state systems to track ongoing work. The Anatomy of agentic memory paper maps this out more formally across memory types.

These are often discussed independently, but they are solving related problems. They decide what remains active, what gets stored, what gets compressed, and what gets forgotten. The emerging picture looks less like a single memory system and more like a memory hierarchy.

Context functions as working memory.
Retrieval provides access.
Memory preserves useful abstractions.
Learning changes the system itself.

failure diagnosis

When an agent loses continuity, the failure is often blamed on context length or model capability. The underlying issue is frequently architectural: retrieval that is too weak to surface what matters, no strategy for what should be compressed or persisted, missing state management, and no memory formation pipeline. A larger context window often postpones these failures rather than solving them.

The more useful diagnostic question is not whether the model can still see the information, but what information deserves to survive after the window is gone.
